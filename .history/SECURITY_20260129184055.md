# Security Policy

## 🛡️ Security Overview

SentineLLM is a security-focused middleware designed to protect LLM applications from prompt injections and prevent secret leakage. We take security seriously and appreciate the community's help in identifying vulnerabilities.

## 🔒 Supported Versions

We actively maintain and provide security updates for the following versions:

| Version | Supported          | Status           |
| ------- | ------------------ | ---------------- |
| 0.1.x   | :white_check_mark: | Active Development |
| < 0.1.0 | :x:                | Not Supported    |

## 🚨 Reporting a Vulnerability

**Please DO NOT report security vulnerabilities through public GitHub issues.**

### Responsible Disclosure Process

1. **Private Reporting** (Recommended)
   - Use GitHub's [Security Advisories](https://github.com/Allesterdev/sentinellm/security/advisories/new)
   - This ensures the vulnerability is handled privately until a fix is available

2. **Email Reporting**
   - Send details to: **security@sentinellm.dev** (or your contact email)
   - Use PGP encryption if possible (key available on request)

3. **What to Include**
   - Type of vulnerability (e.g., Prompt Injection bypass, Secret Leakage, RCE)
   - Full paths of source files related to the vulnerability
   - Location of the affected source code (tag/branch/commit/URL)
   - Step-by-step instructions to reproduce the issue
   - Proof-of-concept or exploit code (if possible)
   - Impact assessment and potential exploitation scenarios

### Response Timeline

- **Initial Response**: Within 48 hours
- **Vulnerability Assessment**: Within 5 business days
- **Fix Development**: Depends on severity (critical: 7 days, high: 14 days)
- **Public Disclosure**: After patch is released and users have time to update

### Severity Classification

We follow [CVSS v3.1](https://www.first.org/cvss/calculator/3.1) for severity rating:

| Severity | CVSS Score | Response Time | Examples |
|----------|------------|---------------|----------|
| **Critical** | 9.0 - 10.0 | 24-48 hours | RCE, Auth bypass, Data exfiltration |
| **High** | 7.0 - 8.9 | 3-7 days | Privilege escalation, Prompt injection bypass |
| **Medium** | 4.0 - 6.9 | 7-14 days | XSS, CSRF, Info disclosure |
| **Low** | 0.1 - 3.9 | 14-30 days | Low-impact bugs, Configuration issues |

## 🔐 Security Best Practices for Users

### Deployment Security

1. **Environment Variables**
   ```bash
   # Never commit .env files
   # Use secrets management (AWS Secrets Manager, HashiCorp Vault)
   cp .env.example .env
   chmod 600 .env
   ```

2. **Network Security**
   - Deploy behind a reverse proxy (nginx, Traefik)
   - Enable TLS/SSL for all endpoints
   - Use API Gateway for rate limiting

3. **Container Security**
   ```bash
   # Run as non-root user in Docker
   USER sentinellm:sentinellm
   
   # Scan images regularly
   trivy image sentinellm:latest
   ```

4. **Monitoring & Logging**
   - Enable audit logging
   - Set up SIEM integration
   - Monitor for anomalous patterns

### Configuration Hardening

```python
# config/security.yaml
security:
  # Enable strict mode for maximum protection
  strict_mode: true
  
  # Rate limiting
  rate_limit:
    requests_per_minute: 60
    burst: 10
  
  # Prompt injection detection
  prompt_injection:
    enabled: true
    sensitivity: "high"  # low, medium, high
    
  # DLP configuration
  dlp:
    enabled: true
    scan_responses: true
    patterns:
      - aws_keys
      - github_tokens
      - credit_cards
```
